# JSON 槽位探针

只问缺口。不要发问卷。提问不是为了聊天，而是为了给 app.v1 JSON 的低置信槽位提供确定性。

旧逻辑是“我不知道，所以我问你”。新逻辑是“这个 receipt 槽位还不够确定，所以我用一个场景题逼近可填写的动作短语”。

## 核心范式

```text
读记忆
  -> 标记每个 JSON 槽位置信度
  -> 找出低置信槽位
  -> 为每个问题指定目标槽位
  -> 生成场景化探针
  -> 用回答更新候选 JSON 值
```

每个问题必须服务于一个待填写字段。回答不会改变 JSON 的问题，删掉。

## 槽位优先原则

先补直接显示在 PNG 上、且证据低于中置信的字段。

```text
P0  receipt.energy / decision / stress / collab 任一为空或低置信
P1  receipt.total 没有核心模式，或只是普通人格摘要
P1  receipt.verdict 没有决策 x 压力张力
P2  receipt.type / rarity / typeGlyph 只剩装饰，没有模式依据
P3  mbti 不确定；可直接写 N/A，不必为了 MBTI 单独发问
```

强制规则：

- 当 `receipt.stress` 的证据置信度低于中置信，必须优先问压力场景。
- 当 `receipt.decision` 的证据置信度低于中置信，必须问一次真实选择场景。
- 同一轮最多问 3 个槽位；超过 3 个说明证据太薄，生成低证据版本。

## 生成流程

### 1. 锁定目标槽位

先写清这题要补哪个字段：

```text
目标槽位        receipt.stress
候选值          慢呼吸三次
不确定点        用户受压后是抽离、爆发还是重建秩序
回答用途        决定 stress 行动签和 verdict 的后半句
```

没有目标槽位，不许提问。

### 2. 提取情境锚点

从可访问材料中先抓真实细节：

```text
真实实体        人名、项目名、公司名、作品名、频道名、工具名
具体卡点        拖延、冲突、崩溃、兴奋、反复修改、拒绝推进
用户原话        用户自己的词、隐喻、口头禅、评价方式
行为节点        某次选择、某段协作、某个项目阶段、某次压力反应
```

每个缺口问题必须至少携带 1 个真实锚点。没有锚点时，不要伪装个性化；先请用户给一个最近的真实场景，只做事实收集，不做人格判断。

### 3. 建立候选 JSON 值

先形成一个临时判断，不把它当结论：

```text
候选字段        receipt.decision
候选值          画出边界
支持证据        哪些记忆片段支持它
反证方向        哪类行为会把它改成 写三行再定 / 明早再回
```

问题必须指向候选值或反证方向。不要为了收集资料而问资料。

### 4. 找出矛盾摩擦

优先寻找两个锚点之间的摩擦：

```text
价值自述 vs 实际行为
长期目标 vs 当下逃避
讨厌规则 vs 主动建立规则
渴望连接 vs 拒绝被打扰
高标准 vs 不愿交付
```

矛盾不是审判。矛盾是人格运行方式最亮的切面。

### 5. 生成个性化探针

问题结构：

```text
目标 JSON 槽位
  -> 我看到的锚点
  -> 当前候选值
  -> 其中的矛盾或缺口
  -> 在具体场景中逼近动作选择
```

优先问动作，不问标签。优先问当时发生了什么，不问“你觉得你是什么人”。

## 问题类型

### 假设验证

用于验证候选 JSON 值。

```text
目标槽位：receipt.decision

我看到你在「<项目名>」里反复提到 <用户原话>。这让我初步把 decision 候选写成「画出边界」，但这里还缺一个真实选择瞬间。

当时你真正卡住的一瞬间，更像是：
- 继续拆计划，直到它看起来不会塌
- 先找一个人碰一下，借外部反馈把自己推出去
- 明知道都能做，但身体在用拖延保护你不被这个项目吞掉
```

### 矛盾探测

用于让用户解释两个锚点之间的摩擦。

```text
目标槽位：receipt.stress + receipt.verdict

你一边很看重「<自由/边界/意义>」，一边在「<具体流程或项目>」里表现出很强的秩序洁癖。

当自由和秩序打架时，你第一反应更像是：
- 先把流程压平，平息混乱带来的焦虑
- 先保护自己的节奏，哪怕流程晚一点再整理
- 表面配合流程，私下另开一套自己的系统
```

### 反证挖掘

用于故意探测候选类型的非典型行为。

```text
目标槽位：receipt.type + receipt.rarity

如果当前材料把你推向「<候选类型或模式>」，我反而要找它不成立的地方。

在「<具体场景>」里，你有没有出现过一种很不像这个模式的动作：
- 放弃结构，直接凭情绪或关系做选择
- 明知道不完美，仍然快速交付
- 为了维持现场气氛，暂时牺牲自己的判断
```

### 画像确认

用于确认 `total` 是否咬住整张票。

```text
目标槽位：receipt.total

根据目前的材料，我把 total 候选压成「<8 字以内短语>」。

这个比喻里最不符合你真实状态的是哪一块？
- 它把你说得太冷
- 它把你说得太有控制感
- 它漏掉了你其实很需要被接住的部分
```

画像确认只能在已有足够锚点时使用。不要用漂亮比喻替代证据。

## 选项规则

- 禁止直接套用模板选项。
- 禁止固定 A/B/C/D 题库感。
- 每个选项必须是动作、代价或防御方式，不是人格标签。
- 选项必须嵌入用户自己的实体、词汇、项目或隐喻。
- 选项要典型且极化，不要设计成每个都温和正确。
- 每个选项必须能被压缩成一个 JSON 候选值。
- 允许用户回答“都不是”，但不要把“其他”写成一个懒惰选项。
- 最多 3 个选项；超过 3 个说明问题没有切准。

## 禁止问法

不要问：

```text
你更内向还是外向？
你更理性还是感性？
你更喜欢计划还是随机？
你是 S 还是 N？
压力大时你一般怎么办？
你觉得你是一个自律的人吗？
```

这些问题只会把人推向标签，不会暴露运行方式。

## 好问题自检

每个问题发出前检查：

```text
有真实锚点吗？
有目标 JSON 槽位吗？
有候选 JSON 值吗？
有反证方向吗？
选项是动作而不是标签吗？
语言像访谈者，不像四级听力吗？
回答会改变候选模式或行动签吗？
```

如果回答不会改变判断，这题就是废题，删掉。
